# 生成模式

[English](../modes.md) | [简体中文](modes.md)

本文说明 `v0.1.1` 中用户可选的生成模式，并在适当位置列出仅 CLI 提供的
高级控制。GUI 有意提供较精简、更安全的选项集。

## 布局模式

### `basic_linear`

**用途：** 将一条 NBS 轨道生成为直线红石结构。

**推荐场景：** 小型结构、解析器检查、输出调试，以及只有一条相关轨道的
简单歌曲。

**重要行为：** 所选轨道从世界原点沿配置方向放置。如果多条轨道含有音符，
必须选择轨道 ID。

**限制：** 不保留多轨立体声场，也不是通用的多图层排列模式。

**控制：** 在 GUI 中选择“基础线性布局”并在“布局选项”中设置轨道 ID，或使用
`--layout-mode basic_linear --track-id <id>`。

### `track_based_stereo`

**用途：** 为每个 NBS 图层/轨道提供稳定空间位置。

**推荐场景：** 需要多轨立体声生成，且可预测的轨道级放置比逐音符空间细节
更重要时。

**重要行为：** 最终图层音量和声像决定每条轨道的位置。支持按轨道拆分中央
声像和整轨碰撞处理。GUI 会让碰撞扩散与中央拆分使用自动默认值，只显示常规
距离控制。

**限制：** 同一轨道中的每个音符共享轨道位置，无法像基于音符的立体声布局
一样准确表达逐音符声像变化。

**控制：** 在 GUI 中选择“基于音轨的立体声布局”，或使用
`--layout-mode track_based_stereo`。CLI 用户可通过 `python main.py --help`
所列选项访问高级中央拆分和处理器控制。

### `note_based_stereo`

**用途：** 根据最终音量和声像放置各个音符发声单元。

**推荐场景：** 适用于从逐音符空间化受益的歌曲，是主要预览模式。

**重要行为：** 布局会生成发声单元候选、分配轨道槽位、验证轨道几何和碰撞，
并通过多轮重试处理失败的发声单元。GUI 配置提供安全、均衡、密集和自定义
设置。结果仍进入共享的方块计划/输出管线。

**限制：** 算法采用启发式。复杂歌曲不保证得到理想排列，大型文件可能需要
大量处理时间和内存。不保证具体的最大歌曲尺寸。

**控制：** 在 GUI 中选择“基于音符的立体声布局”，或使用
`--layout-mode note_based_stereo`。高级用户仍可通过 CLI/配置使用候选、
重试、碰撞、性能分析和空间分析控制。

## 输出格式

### `datapack`

**用途：** 生成用于构建结构并运行已启用模块逻辑的完整 datapack。

**推荐场景：** 直接在 Minecraft 世界中构建，特别是需要启动模块或播放辅助
时。

**重要行为：** 输出包含 `pack.mcmeta` 和与版本匹配的函数目录。主结构和
已启用的运行时行为共享同一构建计划。重新生成会替换 nbs2func 管理的构建
函数目录。

**限制：** 构建函数会修改世界。请备份世界，并选择与区块加载需求相符的
构建方式。

**控制：** 在 GUI 中选择“数据包（datapack）/ mcfunction”，或使用
`--output-format datapack`。

### `schem`

**用途：** 生成兼容 WorldEdit 的 `.schem` 方块结构。

**推荐场景：** 通过兼容 WorldEdit/Litematica 的工作流检查或放置主红石结构。

**重要行为：** 原理图写入器使用与 datapack 写入器相同的结构化方块。
`generation_origin` 是默认坐标原点，也可使用 `min_corner`。默认文件名会
保留 Unicode 输入主文件名。

**限制：** 仅原理图输出会排除启动模块和播放辅助，因为其行为需要运行时
命令。召唤的盔甲架和矿车不会作为存活原理图实体嵌入。

**控制：** 在 GUI 中选择“原理图（.schem）”，或使用
`--output-format schem`、`--schematic-output`、`--schematic-name` 和
`--schematic-origin-mode`。

### `both`

**用途：** 通过 `.schem` 放置方块，同时通过 datapack 提供运行时行为。

**推荐场景：** 从原理图放置完整结构，同时保留启动、播放、记分板、召唤、
执行和实体设置逻辑。

**重要行为：** 原理图包含结构和模块方块，并在支持时包含命令方块 NBT。
Datapack 接收仅运行时计划，不会重复音符盒、中继器、支撑方块或其他主布局
结构。

**限制：** 运行时创建的实体实例不是原理图实体。两种输出产物都必须正确
安装和使用。

**控制：** 在 GUI 中选择“两者都生成”，或使用 `--output-format both`。

## Datapack 构建方式

### `simple_chain`

**用途：** 通过直接相连的函数文件执行生成命令，不使用玩家辅助区块加载。

**推荐场景：** 较小构建、测试或已经加载的受控区域。

**重要行为：** 每个文件最多包含 65535 条命令。非最终文件会预留一条命令
直接调用下一部分。不使用玩家传送、窗口划分、区块等待、计划延迟或等待辅助
函数。

**限制：** 不会自动加载远处区块。调用者负责保持整个目标区域加载。

**控制：** 在 GUI 中选择“简单函数链”，或使用
`--datapack-build-style simple_chain`。

`--no-split-functions` 只作为兼容 CLI/配置入口保留，并映射到
`simple_chain`；尽管旧名称如此，大型输出仍会拆分为直接相连的文件。

### `player_tp`

**用途：** 借助在线玩家提供区块加载，在空间窗口中构建大型结构。

**推荐场景：** 跨越许多区块的大型结构。这是 GUI/配置的默认方式。

**重要行为：** 写入器把命令分为窗口和部分，传送已配置的构建玩家，等待
区块，并按计划运行每个部分及后续窗口。高级 CLI/配置值可控制窗口大小、
等待时间、延迟、玩家名和每部分命令数。

**限制：** 已配置的构建玩家必须有效且在线。离开维度、断开连接或中断执行
可能留下不完整构建。

**控制：** 在 GUI 中选择“玩家传送分段构建”，或使用
`--datapack-build-style player_tp` 及可选的高级玩家传送参数。

## 可选运行时模块

### 启动模块

**用途：** 从已配置的命令方块同步启动生成轨道。

**推荐场景：** 多轨布局和播放辅助配置。

**重要行为：** 添加启动单元、标记设置和启动命令方块。相对于轨道方向，
启动原点必须位于音乐原点后方。

**限制：** 运行时行为无法由仅原理图输出表示。

**控制：** 在 GUI 中启用启动模块，或使用 `--enable-starter-module` 及启动
位置/标签选项。

### 播放辅助

**用途：** 提供基于矿车的播放移动和控件。

**推荐场景：** 需要由记分板驱动移动和准备/开始控件的游戏内播放。

**重要行为：** 添加命令方块、记分板状态、矿车召唤与移动命令以及可选按钮。
GUI 要求先启用启动模块。GUI 生成配置中的播放音乐起点与布局原点相同。

**限制：** 需要有效玩家名、命令权限和运行时 datapack 逻辑。GUI 中不能与
仅原理图输出同时使用。

**控制：** 使用“模块”步骤，或使用以 `--enable-playback-assist`、
`--playback-`、`--vehicle-` 和 `--command-module-origin-` 开头的 CLI 选项。

## 速度控制模式

### `none`

**用途：** 禁用速度控制计算和命令。

**推荐场景：** 保持固定红石时序模型且不需要报告。

**重要行为：** 不生成速度报告或 tick-rate 命令。

**限制：** 不会通过 tick rate 补偿歌曲速度差异。

**控制：** 使用 `--tempo-control-mode none`，或在 GUI 中选择“无”。

### `report`

**用途：** 计算建议的 Minecraft tick rate，但不改变世界。

**推荐场景：** 默认安全模式和手动速度规划。

**重要行为：** 报告 NBS 速度、时序模型、建议 tick rate、后端、命令、重置
命令、警告和权限提示。

**限制：** 报告本身不会执行命令。

**控制：** 使用 `--tempo-control-mode report`，或在 GUI 中选择“仅报告”。

### `command`

**用途：** 通过播放开始/重置逻辑应用建议的 tick rate。

**推荐场景：** 所需后端和权限可用的播放辅助配置。

**重要行为：** 依赖播放辅助。开始序列写出 tick-rate 命令；配置后可在播放
结束时恢复为 20 TPS。

**限制：** 较旧配置需要兼容 Carpet 的能力。1.21.1 配置使用其原版命令
配置，但仍需要适当权限，并且默认情况下命令方块/datapack 未必可用。

**控制：** 使用 `--tempo-control-mode command`、
`--tempo-control-backend auto|carpet|vanilla` 和可选的重置/精度设置，或在
GUI 的“模块”步骤中设置。
